# PROJECT HANDOFF: АИС УЗМО

Этот документ нужен, чтобы продолжить разработку проекта в новом чате или у другого агента без потери контекста. Проект уже прошел большую итерацию UX/архитектуры, поэтому главная задача нового агента: **сначала понять текущие решения, затем аккуратно развивать**, а не переписывать систему по догадкам.

## 1. Что это за проект

Веб-приложение на Django для учета заявок, сведений, фотографий и действий пользователей в рамках системы:

**АИС УЗМО** - автоматизированная информационная система учета заявок материального обеспечения.

Организация в интерфейсе:

**ФКУ ЦХиСО ГУ МВД России по Красноярскому краю**

Приложение предназначено для сотрудников и руководства. Интерфейс должен выглядеть серьезно, ведомственно, аккуратно и современно, без маркетинговой стилистики, без лишней декоративности и без "игрушечного" UX.

## 4. Структура приложений

### `apps.directory`

Справочники и фотографии:

- `TerritorialOrgan` - территориальные органы и подчиненные отделы через `parent`.
- `Department` - направления/отделы ЦХиСО.
- `TerritorialOrganPhoto` - фотографии территориального органа.
- `TerritorialOrganPhotoFolder` - папки фотографий, включая вложенные папки.

Важные решения:

- Фотографии и папки имеют мягкое удаление через `is_deleted`.
- Фотографии и папки хранят автора: `created_by`, `updated_by`.
- Есть `created_department`, чтобы ограничивать изменение/удаление фото и папок отделом автора.
- Дата фотографии - это `created_at`, пользователь ее не выбирает.
- Поиск фотографий должен искать только по имени файла и описанию, а не по полному пути.
- Папки могут быть вложенными.
- В корне раздела "Фотографии" должны показываться только элементы текущей папки, а не все фото из вложенных папок.

### `apps.requests_app`

Основные таблицы, заявки, группировки, экспорт, фотографии заявок.

Ключевые файлы:

- `models.py` - модели заявок и состояний.
- `registry.py` - конфигурация таблиц, вкладок, отделов.
- `views.py` - большая часть CRUD, фильтров, фотографий, экспортов.
- `forms.py` - формы таблиц.
- `permissions.py` - права пользователя по отделам и органам.
- `templates/partials/table_data.html` - отрисовка таблиц.
- `templates/partials/tables_panel.html` - панель вкладок отдела.
- `templates/partials/photos.html` - раздел фотографий.

Важные решения:

- Статуса `Новая` больше нет. Новая заявка сразу создается как `В работе`.
- `NeedStatus` содержит только:
  - `in_work` - В работе
  - `done` - Исполнена
  - `rejected` - Отклонена
- История статусов хранится через универсальную `RequestStatusHistory`.
- В таблицах кнопка истории должна отображаться только когда история действительно есть.
- Последний столбец таблиц - действия. В шапке используется иконка `bi-three-dots-vertical`, а не текст.
- В последнем столбце есть действия: редактирование, удаление, история статуса, прикрепленные фотографии.
- Массовое удаление заявок пока не делалось и не должно появляться случайно.

### `apps.audit`

Журнал действий:

- `AuditLog` хранит действие, пользователя, объект, old/new values, территориальный орган, IP, user-agent.
- `views.py` готовит человекочитаемые события через `prepare_log`.
- `audit_log` - полный журнал для администратора.
- `my_audit_log` - журнал действий в пределах доступных пользователю отделов/органов.
- `middleware.py` хранит current request и обновляет `last_seen_at` при обычных запросах.

Важные решения:

- Журнал должен быть понятен начальнику, не программисту.
- Не показывать технические ID, технические имена моделей и лишние поля, если они не нужны пользователю.
- Событие должно быть главным смысловым текстом.
- Столбец "Действие" был убран, потому что событие само объясняет действие.
- В подробностях не нужно показывать лишние блоки вроде "Дополнительные параметры отсутствуют".
- Скачивания фотографий/архивов не логируются, чтобы не засорять журнал.
- Прикрепление/открепление фотографий, изменение статуса, изменения ТМЦ должны логироваться как понятные события.

### `apps.accounts`

Пользователи, профили, активация, админ-панель.

Ключевые файлы:

- `models.py` - `UserProfile`.
- `admin.py` - кастомизация стандартной Django admin для пользователей.
- `views.py` - активация, кастомная админ-панель, presence heartbeat.
- `forms.py` - активация учетной записи.
- `templatetags/account_tags.py` - отображение ФИО.

Важные решения:

- Пользователь имеет стандартную Django `User` + `UserProfile`.
- В `User` хранятся:
  - `last_name`
  - `first_name`
  - `username`
  - пароль
- В `UserProfile` хранятся:
  - `middle_name`
  - `role`
  - `allowed_organs`
  - `allowed_departments`
  - `activation_code`
  - `last_seen_at`
- В интерфейсе где возможно показывать не логин, а ФИО/инициалы.
- В бургер-меню можно показывать полное имя и отчество.
- Администратор создает пользователя без пароля и выдает логин + код активации.
- Пользователь на странице входа переходит в "Активация учетной записи", вводит логин и придумывает пароль.

Роли:

- `admin` - администратор.
- `operator` - оператор.
- `observer` - наблюдатель.

## 5. Главная страница

Главная страница состоит из трех рабочих блоков:

1. Территориальные органы.
2. Отделы.
3. Рабочая область с таблицами/фотографиями.

Ключевые UX-решения:

- На широком экране блоки располагаются горизонтально.
- На среднем масштабе/ширине возможен вариант: два блока сверху, третий ниже.
- На узкой ширине блоки складываются вертикально.
- На больших экранах блоки стараются уложиться в видимую высоту.
- При увеличении масштаба внутренние скроллы должны исчезать, чтобы не было двойных скроллов.
- Подвал должен идти после контента, не красть место у рабочих блоков.
- Первый блок может работать в режиме:
  - один территориальный орган;
  - выборочно несколько территориальных органов.
- При переключении из "Выборочно" обратно в одиночный режим нужно восстанавливать орган, выбранный до мультивыбора.
- При сбросе фильтров в таблице нельзя случайно сбрасывать выбранный территориальный орган на первый.
- Подсветка выбранного территориального органа и отдела должна быть понятной, без галочки справа.
- Подчиненные отделы в поиске органов раньше участвовали, но подсказка теперь "Поиск по территориальным органам".

Сворачивание блоков:

- Блок "Отделы" можно свернуть в блок "Территориальные органы".
- Блок "Территориальные органы" можно свернуть отдельно.
- Если оба свернуты, остается одна плавающая полупрозрачная кнопка.
- Если свернут только блок "Территориальные органы", кнопку разворота лучше показывать в блоке "Отделы", а не плавающей кнопкой.

Не ломать эту логику: она долго выверялась под разные масштабы браузера.

## 6. Таблицы и отделы

### Обеспечение товарно-материальными ценностями

Отдел: `tmc`

Вкладка: `Заявка`

Особенности:

- Одна заявка может содержать много позиций ТМЦ.
- Структура таблицы должна соответствовать ТЗ:
  - объединенный заголовок "Сведения о потребности ТМЦ";
  - под ним "Наименование", "Количество";
  - объединенный заголовок "Заявка";
  - под ним "Номер", "Дата", "Исполнение заявки";
  - "Описание";
  - действия.
- Столбцы блока "Заявка" центрируются.
- Заявки визуально разделяются жирными линиями.
- При наведении на любую позицию ТМЦ подсвечивается вся заявка.
- Есть группировки:
  - по заявкам;
  - по ТМЦ;
  - по дате;
  - по территориальному органу, только если выбрано несколько органов.
- В режиме "По ТМЦ" статусные плашки заявок не нужны.
- В режиме "По ТМЦ" экспорт должен экспортировать именно группировку ТМЦ, а не обычные заявки.
- Есть справочник ТМЦ (`TmcProduct`).
- При вводе позиции заявки работает умный поиск товаров по токенам и похожести.
- Если пользователь не выбрал подсказку, но ввел ровно существующее название, система должна связать с существующим товаром.
- Если товар не найден, он автоматически создается в справочнике.
- Если позиция не добавлена, форма должна показать ошибку, но не ломать возможность добавить позицию после ошибки.

Отдел: `fire`

Вкладки:

- `Огнетушители`
- `Пожарная сигнализация`
- `Охранная сигнализация`

- Сроки годности не должны красить всю строку грубо. Используются аккуратные предупреждающие плашки/маркеры в стиле статусов.
- Заявка пожарной безопасности работает по аналогии с другими заявочными таблицами.

## 7. Фотографии

Раздел фотографий открывается из блока выбранного территориального органа.

Ключевые возможности:

- загрузка одной или множества фотографий;
- drag and drop по всей области блока;
- при перетаскивании на папку загрузка идет в эту папку;
- лимит массовой загрузки: до 300 фото за один раз;
- предпросмотр фотографий в окне массовой загрузки;
- папки в виде карточек, проводниковая логика;
- вложенные папки;
- переименование папок;
- перемещение папок;
- удаление папок мягкое;
- скачивание отдельной фотографии;
- скачивание всех фотографий;
- скачивание папки архивом;
- подготовка архива не должна ломать переходы между разделами;
- есть lightbox с zoom/pan/стрелками;
- фотография в lightbox не должна обрезаться по краям;
- поиск и сортировка работают на сервере;
- сброс фильтров фотографий сбрасывает поиск/сортировку, но не должен выкидывать из текущей папки;
- путь папок кликабельный, с иконками количества фото/папок;
- в карточке папки количество фотографий и папок отображается иконками, не простым текстом;
- описание фотографии ограничено 3 строками с корректным ellipsis.

Важные подводные камни:

- Не показывать в корне фотографии из вложенных папок.
- При перемещении фотографии/папки показывать полный путь папки, иначе одинаковые названия папок запутают пользователя.
- Кнопки на карточках фото прижаты вниз.
- Кнопки на карточке папки не должны сжимать название папки.
- Нажатие на кнопки папки не должно проваливать пользователя внутрь папки.
- Нельзя физически удалять файлы с сервера при обычном удалении - используется мягкое удаление.
- Скачивание больших папок может долго готовить архив. Для архива используется потоковый/дисковый подход, чтобы не держать все в памяти.

## 8. Прикрепленные фотографии к заявкам

Есть модель `RequestPhotoLink`.

Логика:

- Фото прикрепляются к заявке при создании или редактировании через модальное окно выбора фото.
- В последнем столбце заявки есть кнопка прикрепленных фотографий, но если фото нет, кнопку можно не показывать.
- Количество прикрепленных фото показывается в tooltip, например `Прикрепленные фотографии (N шт.)`.
- В окне прикрепленных фото можно:
  - посмотреть прикрепленные фото;
  - скачать все;
  - прикрепить еще;
  - открепить фото.
- Прикрепление должно поддерживать поиск, фильтрацию и пагинацию.

## 9. Журнал действий

URL:

```text
/audit/     полный журнал администратора
/audit/my/  журнал доступных пользователю отделов
```

Дизайн:

- Таблица в стиле приложения.
- Пагинация такая же по логике, как в таблицах.
- Шапка таблицы закреплена.
- Событие отображается как основной смысловой текст.
- Действие отображается как аккуратная цветная плашка в стиле основных статусов.
- Не использовать чрезмерный `font-weight: 800`; лучше 500-600.
- Кнопка подробностей называется `Открыть`.

Фильтры:

- пользователь;
- действие;
- объект;
- отдел;
- территориальный орган;
- даты.

Ранее поиск по журналу был признан почти бесполезным и был убран.

Важные UX-решения:

- Обычный пользователь видит журнал действий людей из доступных ему отделов/органов, а не только свои действия.
- Администратор видит полный журнал.
- Для пользователя фильтры должны быть проще, но сейчас фильтр по людям и отделам допустим, потому что он может смотреть доступные отделы.
- В подробностях не надо показывать "Отдел автора" для фотографий/папок, если это выглядит лишним.
- Для событий с папкой писать `Папка фотографий`, а не `Папка с фотографиями`.
- Для удаления писать просто `Запись удалена`, без фраз вроде "скрыта из рабочих разделов".

## 10. Пользователи, доступы и безопасность

Пользователь может иметь доступ к нескольким отделам и нескольким территориальным органам.

Права:

- Администратор имеет полный доступ.
- Оператор может создавать/редактировать/удалять только в закрепленных отделах и органах.
- Наблюдатель смотрит, но не меняет.

Фотографии:

- Пользователь может изменять/удалять фото и папки только если они созданы человеком из его отдела.
- Не нужно делать сложную проверку "папка содержит чужое" при удалении, если загрузка в чужие папки запрещена и права корректно ограничены по `created_department`.

Первый вход:

- Администратор создает учетную запись без пароля.
- Система генерирует 6-значный цифровой код активации.
- Пользователь вводит логин и код, затем придумывает пароль.
- На странице входа ссылка называется `Активация учетной записи`.
- На странице активации заголовок `Регистрация`.
- Поле пароля называется `Придумайте пароль`.

Онлайн-статус:

- Браузер авторизованного пользователя отправляет heartbeat на `/accounts/presence/`.
- Интервал heartbeat: 30 секунд.
- Онлайн-окно: 1 минута.
- Админ-панель обновляется раз в 30 секунд.
- Закрытие вкладки/браузера не считается выходом из системы.
- Событие `Выход` логируется только при нажатии `Выйти из системы`.
- Если вкладка закрыта, heartbeat прекращается, и примерно через минуту пользователь станет офлайн.

## 11. Кастомная админ-панель

URL:

```text
/control/
```

Файлы:

```text
apps/accounts/views.py
templates/admin_panel/index.html
templates/admin_panel/_panel.html
static/css/app.css
static/js/app.js
```

Решение:

- Стандартная Django admin на `/admin/` остается как технический резерв.
- Пункт `Панель администратора` в бургер-меню ведет на `/control/`, а не на `/admin/`.
- Внутри кастомной панели есть кнопка `Таблицы БД`.
- Панель обновляет внутренний фрагмент через HTMX polling раз в 30 секунд.
- При HTMX-запросе сервер возвращает только `admin_panel/_panel.html`, не всю страницу.
- Это важно: иначе можно случайно отрисовать футер/шапку внутри панели.

Сейчас панель показывает:

- сотрудников;
- онлайн сейчас;
- ожидающих активации;
- территориальные органы;
- заявки в работе;
- события сегодня;
- сотрудников и доступы;
- активацию;
- кто сейчас в системе;
- доступ по отделам;
- контур данных;
- последние события.

Что планировалось дальше:

- сделать полноценный современный экран управления сотрудниками;
- сделать экран управления доступами;
- добавить аналитические блоки для руководства:
  - органы с наибольшим количеством заявок;
  - заявки в работе по отделам;
  - просроченные/проблемные заявки;
  - динамика по датам;
  - частые позиции ТМЦ;
  - активность пользователей;
  - состояние заполненности данных по органам;
- не использовать слово `Топ`; лучше `Наибольшая активность`, `Сводка`, `Рейтинг по количеству заявок`, `Наибольшее количество заявок`, если нужно официальнее.

Важно:

- Не превращать админ-панель в копию стандартной CRUD admin.
- Руководству нужен обзор и контроль, а технические формы - вторым уровнем.
- Все контролы должны быть в стиле проекта.
- Не добавлять стандартные браузерные select/input без стилизации.

## 12. CSS и UI-правила

Основной CSS:

```text
static/css/app.css
```

Важно:

- Там много логики адаптивности, особенно главной страницы.
- Не делать широкие стилистические рефакторы без необходимости.
- Не удалять правила `:has(...)`, если не понимаешь, зачем они нужны.
- После изменения CSS поднять query-string версию в `templates/base.html`.
- Сейчас версия CSS/JS подключается вручную через `?v=...`.

Стиль:

- светлый, голубовато-белый ведомственный стиль;
- аккуратные панели;
- небольшие радиусы, обычно до 8px;
- мягкие плашки;
- без темных dashboard-экранов;
- без маркетинговых hero-блоков;
- без декоративных градиентных сфер/орбов;
- без огромной типографики внутри рабочих панелей;
- шрифт не делать слишком крупным;
- важная цель: все должно влезать при разных масштабах браузера.

Кнопки:

- используются Bootstrap Icons.
- Кнопки действий в таблицах и фото часто иконками.
- Tooltip должен быть аккуратным и не должен создавать горизонтальный скролл.
- Раньше были проблемы с tooltip при zoom браузера, поэтому не внедрять сложные позиционируемые тултипы без проверки.

Таблицы:

- `table-wrap` и высотная логика очень чувствительны.
- Шапки таблиц закреплены.
- На некоторых масштабах внутренние скроллы должны исчезать.
- Не добавлять лишние `max-height`/`height: 100%` без проверки на ноутбуке и большом мониторе.

Модальные окна:

- При открытом модальном окне фон не должен прокручиваться колесом.
- При этом кастомный внешний скроллбар не должен исчезать/скакать.
- В модалках формы должны быть без вложенных `<form>` внутри `<form>`. Такая ошибка уже ломала сохранение.

## 13. JavaScript

Основной файл:

```text
static/js/app.js
```

Что там есть:

- хранение выбранного органа/режима/отдела в storage;
- HTMX hooks;
- кастомные select;
- модалки;
- tooltips;
- таблицы;
- фотографии;
- lightbox;
- bulk upload;
- pagination jump;
- presence heartbeat.

Важные функции/константы:

- `PRESENCE_HEARTBEAT_MS = 30000`
- `startPresenceHeartbeat()`
- `sendPresencePing()`
- `BULK_PHOTO_BATCH_SIZE = 25`
- `ORGAN_MODE_KEY`
- `MULTI_ORGANS_KEY`
- `TABLE_STATE_PREFIX`

Подводные камни:

- HTMX-запросы включают верхнюю полосу загрузки. Presence heartbeat специально сделан через `fetch`, чтобы не дергать эту полосу.
- Не все запросы должны сохранять состояние таблиц.
- Сброс фильтров должен аккуратно чистить только состояние текущей таблицы, а не ломать выбранный орган.
- Много логики завязано на `data-*` атрибуты в шаблонах.

## 14. Экспорт

Экспорт таблиц должен учитывать текущий режим:

- обычные заявки;
- группировка по ТМЦ;
- группировка по дате;
- группировка по территориальному органу;
- множественный выбор территориальных органов.

Подводный камень:

Раньше была ошибка: в режиме выбора нескольких органов таблица показывала найденные записи со столбцом территориального органа, а экспорт выгружал больше записей и без нужного столбца. Экспорт должен соответствовать тому, что видит пользователь.

Для больших файлов:

- архивы фотографий и Excel-экспорт лучше готовить через временный файл на диске, а не держать все в RAM.

## 15. Производительность

Сейчас проект рассчитан на ведомственное локальное/корпоративное использование, но уже есть защита от очевидных проблем:

- фотографии пагинируются;
- массовая загрузка ограничена 300 фото за раз;
- bulk upload отправляет файлы пакетами;
- поиск фотографий идет на сервере;
- поиск таблиц должен срабатывать без потери фокуса;
- админ-панель обновляется фрагментом, а не всей страницей.

Если добавлять аналитику:

- не считать тяжелые агрегаты в циклах по каждой строке;
- использовать ORM aggregation;
- не делать графики/метрики, которые на каждом polling будут выполнять десятки тяжелых запросов;
- при необходимости кэшировать или обновлять только отдельные блоки.

```text
Кастомная админ-панель находится на /control/.
После изменений запускай python manage.py check и релевантные тесты.
Если видишь странную реализацию, сначала проверь handoff и существующие тесты: возможно, это осознанное UX-решение.
```

Плохой промпт:

```text

```

Такой промпт опасен: новый агент может удалить важную адаптивную логику, упростить права, сломать таблицы, заменить кастомные контролы стандартными или переписать существующий UX.

## 20. Главные "не ломать"

1. Не ломать трехколоночную главную страницу и ее адаптивность.
2. Не ломать режим "Один/Выборочно" для территориальных органов.
3. Не возвращать статус заявки `Новая`.
4. Не показывать скрытую вкладку `Автотранспорт`, если пользователь не попросил.
5. Не выводить фотографии из вложенных папок в корне.
6. Не удалять физически фотографии при обычном удалении.
7. Не делать стандартные некрасивые select/input там, где уже есть кастомный стиль.
8. Не показывать технические поля в журнале действий для руководства.
9. Не превращать `/control/` в копию `/admin/`.
10. Не менять интервал heartbeat и онлайн-окно без явного решения.
11. Не делать массовые удаления без отдельного обсуждения.
12. Не использовать слово `Топ` в интерфейсе руководства без причины; стиль должен быть официальнее.
13. Не делать большие CSS/JS refactor вместе с функциональными изменениями.
14. Не откатывать чужие незакоммиченные изменения.

## 21. Рекомендуемый порядок развития админ-панели

1. Экран/блок "Сотрудники":
   - список сотрудников;
   - статус онлайн;
   - роль;
   - отделы;
   - территориальные органы;
   - состояние активации;
   - быстрый переход к редактированию.

2. Экран/блок "Доступы":
   - сотрудники по отделам;
   - сотрудники по территориальным органам;
   - предупреждения о пользователях без отдела/органа;
   - массовое назначение доступа, но осторожно.

3. Аналитика руководителя:
   - количество заявок в работе по отделам;
   - количество исполненных/отклоненных;
   - органы с наибольшим количеством заявок;
   - динамика по датам;
   - частые позиции ТМЦ;
   - проблемные/долго не меняющиеся заявки.

4. Состояние данных:
   - где давно не обновлялись состояния;
   - где нет последних записей по состояниям;
   - где мало/нет фотографий;
   - где есть ожидающие активации сотрудники.

5. Только после этого:
   - перенос части функций из стандартной Django admin в кастомный UI.

## 22. Если что-то кажется странным

Перед изменением проверь:

- историю в `git log`;
- тесты;
- `PROJECT_HANDOFF.md`;
- существующие шаблоны;
- связанные CSS/JS data-атрибуты.

Многие "странности" появились после конкретных UX-багов:

- двойные скроллы при масштабировании;
- tooltip уезжал при zoom;
- модалка разрешала скроллить фон;
- форма ломалась из-за вложенной формы;
- кастомный select дублировался нативным;
- сброс фильтров сбивал выбранный орган;
- экспорт не совпадал с режимом таблицы;
- фото из вложенных папок показывались в корне.

Не надо "упрощать" эти места без проверки.
