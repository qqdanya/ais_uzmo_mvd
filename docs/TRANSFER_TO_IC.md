# Передача исходного кода в информационный центр

Этот документ предназначен для разработчика, который готовит комплект для ИЦ.
Сотрудникам ИЦ передаются четыре инструкции из релиза:

- `DEPLOY_LINUX.md` — быстрый штатный вариант через установщик;
- `DEPLOY_LINUX_MANUAL.md` — ручное развёртывание с командами для каждого этапа;
- `QA_CHECKLIST.md` — приёмка установленной системы;
- `MAINTENANCE.md` — проверка, обновление и резервное копирование после установки.

Для установки администратор выбирает один из первых двух вариантов и не смешивает
их на одном сервере. Приёмка и сопровождение выполняются по двум следующим
инструкциям.

## Какой способ выбрать

Сначала согласуйте способ передачи с ответственным за информационную безопасность.
По одному названию программы нельзя определить допустимый канал передачи.

Рекомендуемый порядок:

1. Если в ИЦ есть внутренний GitLab, Gitea или другой ведомственный Git-сервер —
   создать там новый закрытый репозиторий из проверенного релизного архива.
2. Если внутреннего Git-сервера нет — передавать нумерованный релизный архив
   через разрешённый защищённый канал или учтённый носитель.
3. Private-репозиторий GitHub технически позволяет `clone` и `pull`, но использовать
   внешний GitHub можно только после разрешения ответственного за ИБ.

Для данного проекта базовый вариант передачи — **релизный архив**, а не доступ к
личному GitHub разработчика.

Не переносите в ИЦ каталог `.git` и полную историю личного репозитория. Релизный
архив содержит только актуальные исходники и не содержит историю разработки или
сведения о прежних развёртываниях. Если ИЦ нужен Git для дальнейшей работы, новый
репозиторий создаётся уже у них из этого архива.

## Что передать

Основной комплект:

```text
ais_uzmo-1.0.0.tar.gz   исходный код одной зафиксированной версии
SHA256SUMS              контрольная сумма архива
RELEASE.txt             версия, дата и номер коммита
```

В архиве уже должны находиться инструкции `docs/DEPLOY_LINUX.md`,
`docs/DEPLOY_LINUX_MANUAL.md`, `docs/QA_CHECKLIST.md` и
`docs/MAINTENANCE.md`.

Для установки нужны пакеты Ubuntu и Python. Если внешнего доступа нет, в ИЦ
должны быть внутренние зеркала APT и PyPI. Отдельный `wheelhouse` заменяет только
PyPI, но не системные пакеты Ubuntu. В передаваемом `wheelhouse` обязательны
собственный `SHA256SUMS` и все пакеты из `requirements.txt`; каталог готовится под
точные Ubuntu, архитектуру и версию Python. Полностью автономный комплект также
должен содержать согласованный источник системных пакетов Ubuntu и проверяться на
отдельной машине с теми же параметрами.

SHA-256 нужно подтвердить по независимому согласованному каналу либо подписать
комплект принятой в организации электронной подписью. Передача архива и хеша из
одного непроверенного источника подтверждает целостность, но не автора файлов.

## Что передавать отдельно

Не смешивайте код, настройки и рабочие данные.

- Реальный `.env`, пароли, ключи и сертификаты передаются отдельно по согласованному
  защищённому каналу. Для новой установки старый `.env` вообще не нужен.
- Если требуется перенести существующую систему, не копируйте работающую базу,
  каталог данных PostgreSQL, живые `media` или `runtime`. Сначала создайте штатную
  резервную копию. Передавать можно только завершённый каталог с `COMPLETE` и
  успешно проверенным `SHA256SUMS`, зашифрованный разрешённым способом.
- Вместе с каждой резервной копией сохраняйте релизный архив той версии, которая
  указана в файле `RELEASE` внутри копии, а также его `SHA256SUMS` и `RELEASE.txt`.
  Резервная копия содержит данные и настройки, но не исходный код приложения.
- Демонстрационные данные предварительно очищаются от реальных персональных данных.

Обычная электронная почта, мессенджер, публичное облако и неучтённая флешка для
такой передачи не подходят без прямого разрешения ответственного за ИБ.

## Подготовка версии

Перед выпуском релиза должны быть завершены проверки, а нужные изменения —
закоммичены. При незакоммиченных или новых файлах скрипт остановится и не создаст
архив.

Укажите номер в `deploy/VERSION`, сохраните все изменения коммитом, создайте тег
той же версии и запустите готовую сборку. Пример для версии `1.0.0` в Git Bash:

```bash
git status --short
python manage.py check
python manage.py makemigrations --check --dry-run
python manage.py test --parallel 4
python scripts/refactor_static_check.py
for script in deploy/*.sh; do bash -n "$script" || exit 1; done
git tag -a v1.0.0 -m "AIS UZMO 1.0.0"
bash deploy/release.sh
```

Все проверки должны завершиться успешно, а `git status --short` перед созданием
тега не должен ничего выводить. `deploy/release.sh` повторно выполняет
автоматические проверки и остановится, если есть несохранённые изменения, версия
не совпадает с тегом или тег указывает не на текущий коммит. Готовые три файла
появятся в каталоге `dist`.

Проверьте собранный комплект:

```bash
(cd dist && sha256sum -c SHA256SUMS)
tar -tzf dist/ais_uzmo-1.0.0.tar.gz
```

Убедитесь, что в архиве есть `apps`, `config`, `templates`, `static`, `deploy`,
`manage.py`, `requirements.txt` и четыре инструкции из раздела «Что передать».

До передачи обязательно проверьте архив на отдельной чистой виртуальной машине
Ubuntu 24.04: выполните штатную установку из архива и `deploy/check.sh`. Для
релиза-обновления дополнительно проверьте переход с предыдущей поддерживаемой
версии через `deploy/update.sh`. Виртуальная машина не должна содержать рабочие
данные и не должна быть боевым сервером.

## Чего не должно быть в архиве исходного кода

```text
.git/
.env и другие заполненные файлы окружения
db.sqlite3 и любые выгрузки баз данных
media/ и реальные фотографии
runtime/
dashboard_thresholds.json и dashboard_thresholds.json.tmp
staticfiles/
.venv/ и venv/
logs/ и *.log
приватные ключи, токены, пароли и сертификаты
.claude/ и другие локальные файлы разработчика
```

Команда `git archive` берёт только файлы из выбранного коммита или тега. Это
защищает от случайного попадания локальных `.env`, базы, фотографий и других
незакоммиченных файлов, но не заменяет ручную проверку содержимого архива.
Перед выпуском также проверьте текущие файлы и историю репозитория на секреты и
реальные персональные данные: проверка имён файлов в `release.sh` не обнаруживает
пароль, записанный внутри обычного исходного файла.

## Если репозиторий раньше был публичным

1. Переключить его в private как можно раньше.
2. Проверить всю историю Git на пароли, `.env`, ключи, базы и персональные данные.
3. Любой секрет, который когда-либо попадал в публичную историю, считать
   раскрытым и заменить.
4. Помнить, что перевод репозитория в private не удаляет уже сделанные копии и
   не делает приватными ранее созданные публичные форки.

Уже установленное приложение после этого не остановится. Штатная установка и
обновление выполняются релизными архивами без `git pull`, поэтому серверу ИЦ
доступ к GitHub не нужен.

## Короткий лист передачи

Зафиксируйте в сопроводительном листе:

- название и номер версии;
- дату передачи;
- имя архива и SHA-256;
- результат проверки на чистой Ubuntu 24.04;
- наличие или отсутствие отдельного комплекта данных;
- место хранения соответствующего релизного архива для восстановления;
- кому и каким разрешённым способом передан комплект;
- контакт разработчика для вопросов по установке.
