# Финальная проверка разработчика

Этот чеклист выполняется разработчиком только в локальном или CI-окружении до
создания релиза. Это не серверная инструкция: команды ниже нельзя запускать на
рабочей базе или в каталоге установленного приложения.

## 1. Автоматические проверки

В подготовленном локальном виртуальном окружении выполнить:

```bash
python manage.py check
python manage.py makemigrations --check --dry-run
python manage.py test --parallel 4
python scripts/refactor_static_check.py
```

Все команды должны завершиться успешно. Если модели изменялись,
`makemigrations --check --dry-run` должен подтвердить, что нужные миграции уже
созданы и добавлены в исходный код.

Эти проверки не требуют подключения к рабочей базе. Не подставляйте в локальный
`.env` адреса, пароли или данные установленной системы.

## 2. Проверка изменений

```bash
git diff --check
git status --short
```

- Просмотреть полный diff и убедиться, что он соответствует задаче.
- Проверить новые и изменённые миграции.
- Убедиться, что отладочный код, временные сообщения и локальные пути удалены.
- Проверить, что зависимости изменены одновременно в `requirements.in` и
  зафиксированном `requirements.txt`, если это требуется.
- Не добавлять реальные секреты, базы, фотографии, журналы и локальные настройки
  в Git.

## 3. Функциональная проверка в локальном окружении

Для затронутых функций проверить как минимум:

- вход и выход пользователя;
- открытие главного экрана и нужных разделов;
- создание, изменение статуса и удаление контрольной заявки;
- загрузку, просмотр, скачивание и удаление контрольной фотографии;
- административные фильтры и журнал действий;
- ограничения для всех затронутых ролей;
- отображение интерфейса, модальных окон, значков, графиков и сообщений об
  ошибках.

Использовать только искусственные локальные данные.

## 4. Проверка deployment-изменений

Если менялись production-настройки или файлы в `deploy`, дополнительно проверить
согласованность:

- `install.sh`, `update.sh` и `check.sh`;
- systemd units и Nginx-шаблона;
- создания `.env`, PostgreSQL, Redis, статических и загружаемых файлов;
- резервного копирования и безопасного поведения при ошибке;
- `DEPLOY_LINUX.md`, `DEPLOY_LINUX_MANUAL.md` и `MAINTENANCE.md`.

Не проверяйте это командами на установленной рабочей системе. Используйте
одноразовое изолированное окружение с той же поддерживаемой версией Ubuntu либо
автоматизированную CI-проверку.

## 5. Готовность к релизу

- Все нужные изменения и миграции сохранены в коммитах.
- Номер следующей версии определён.
- Рабочая папка не содержит случайных или несохранённых файлов.
- Канонические инструкции соответствуют выпускаемому коду.

После выполнения этого чеклиста перейти к
[`DEPLOY_CHECKLIST.md`](DEPLOY_CHECKLIST.md). Релизный архив создаётся только
скриптом `deploy/release.sh`; ручной список исключений его не заменяет.
