# АИС учёта заявок на материальное обеспечение

Веб-приложение на Django для учёта заявок, сведений о материальной базе, фотографий территориальных органов и действий пользователей.

Система предназначена для внутреннего использования сотрудниками ФКУ ЦХиСО ГУ МВД России по Красноярскому краю.

> **Сотруднику ИЦ:** выберите штатную
> [`установку через готовый скрипт`](docs/DEPLOY_LINUX.md) или
> [`ручное развёртывание`](docs/DEPLOY_LINUX_MANUAL.md), затем пройдите
> [`приёмку`](docs/QA_CHECKLIST.md). Сопровождение установленной системы:
> [`docs/MAINTENANCE.md`](docs/MAINTENANCE.md).

## Возможности

- Авторизация пользователей и активация учётных записей.
- Разграничение доступа по ролям, территориальным органам и отделам.
- Учёт заявок по направлениям: ТМЦ, транспорт, пожарная безопасность, антитеррор, ЦИТСиЗИ, УОТО.
- Контроль уникальности номера заявки в рамках территориального органа и отдела.
- Учёт нескольких ответов по каждой заявке: самый новый считается итоговым на данный момент, предыдущие — промежуточными, без автоматического изменения статуса заявки.
- Статусы заявок: «В работе», «Исполнена», «Отклонена».
- История изменения статусов и журнал действий пользователей.
- Загрузка, хранение, просмотр и прикрепление фотографий к заявкам.
- Экспорт данных и подготовка архивов фотографий.
- Административная панель для управления сотрудниками, правами, справочниками, заявками и настройками.

## Технологии

- Python 3.12+ / Django 5.2
- PostgreSQL для серверного развёртывания
- SQLite для локальной проверки и разработки
- Django Templates, HTMX, Bootstrap 5
- Pillow, openpyxl
- Gunicorn, Nginx, WhiteNoise

## Быстрый запуск на Windows для проверки

```powershell
python -m venv .venv
.\.venv\Scripts\activate
python -m pip install -r requirements.txt
copy .env.example .env
python manage.py migrate
python manage.py seed_initial_data
python manage.py test --parallel 4
python manage.py runserver 127.0.0.1:8000
```

После запуска открыть:

```text
http://127.0.0.1:8000/
```

Подробная инструкция: [`docs/RUN_WINDOWS.md`](docs/RUN_WINDOWS.md).

## Серверное развёртывание

Production-вариант:

```text
Ubuntu 24.04 + PostgreSQL + Redis + Gunicorn + Nginx + ежедневные резервные копии
```

Для ИЦ программа передаётся нумерованным релизным архивом вместе с SHA-256.
Готовый установщик сам настраивает службы, базу, кэш, резервные копии и сетевые
ограничения. Простая инструкция: [`docs/DEPLOY_LINUX.md`](docs/DEPLOY_LINUX.md).

Базовый установщик открывает HTTP только для указанной внутренней сети. VPN или
ведомственный защищённый контур определяет ИЦ. Если обязателен HTTPS, требуется
отдельный согласованный комплект с поддержкой TLS в установке, проверке и
обновлении.

Порядок безопасной подготовки и передачи комплекта разработчиком:
[`docs/TRANSFER_TO_IC.md`](docs/TRANSFER_TO_IC.md).

## Переменные окружения

Для локального запуска используется шаблон:

```text
.env.example
```

При штатной серверной установке закрытый `/srv/ais_uzmo/.env` создаёт
`deploy/install.sh`; вручную копировать в него пример не нужно. Файл
`.env.production.example` служит справочником разработчика для нестандартного
согласованного развёртывания.

Реальный `.env` содержит секреты и не должен попадать в Git или релизный архив.

## Зависимости

В проекте используются два файла зависимостей:

```text
requirements.in   прямые зависимости проекта
requirements.txt  lock-файл с точными версиями, созданный через pip-compile
```

Для установки на рабочей станции или сервере использовать:

```bash
python -m pip install -r requirements.txt
```

Обновление зависимостей выполняет разработчик в отдельной ветке:

```bash
python -m pip install pip-tools
python -m piptools compile requirements.in --output-file=requirements.txt
python -m pip install -r requirements.txt
python manage.py test --parallel 4
```

## Начальные данные

```bash
python manage.py seed_initial_data
```

Команда создаёт отделы, территориальные органы, подчинённые подразделения и начального руководителя, если в `.env` заданы `SUPERUSER_USERNAME`, `SUPERUSER_EMAIL`, `SUPERUSER_PASSWORD`.

Повторный запуск команды не создаёт дубликаты.

## Что нельзя хранить в репозитории и передавать в релизном архиве

Не собирайте исходники через проводник и не составляйте собственный список
исключений. `deploy/release.sh` создаёт архив только из зафиксированного Git-тега и
отклоняет реальные `.env`, базы и дампы, `media`, runtime-данные, логи, виртуальные
окружения, ключи и локальные служебные файлы.

Миграции Django, `requirements.in`, `requirements.txt`, безопасные примеры `.env`,
`static/vendor/` и документацию из исходного кода удалять не нужно.
